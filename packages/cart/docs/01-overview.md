# Cart

## Purpose

The `aiarmada/cart` package is the cart foundation for the Commerce monorepo. It owns cart persistence, cart items, conditions, metadata, and cart lifecycle events.

## What this package owns

- Database-backed cart storage and optimistic locking
- Cart items, cart conditions, metadata, and dynamic condition support
- Cart lifecycle flows such as merge, clear, add, remove, and login migration
- Optional owner-aware cart persistence through `commerce-support`

## What this package does not own

- Recovery campaigns, alerting, and behavioral analytics; those belong to `aiarmada/signals`
- Checkout orchestration; that belongs to `aiarmada/checkout`
- Voucher, promotion, shipping, or tax domain rules; those integrate with cart but are owned elsewhere
- Filament admin surfaces; those belong to `aiarmada/filament-cart`

## Related packages

- [`aiarmada/filament-cart`](../../filament-cart/docs/01-overview.md) — Filament admin views for cart snapshots and operations
- [`aiarmada/checkout`](../../checkout/docs/01-overview.md) — checkout orchestration on top of cart state
- [`aiarmada/signals`](../../signals/docs/01-overview.md) — analytics and alerting layer for cart events
- [`aiarmada/commerce-support`](../../commerce-support/docs/01-overview.md) — owner scoping and shared helpers

## Main models services or surfaces

- **Core surfaces** — cart storage, items, conditions, metadata, and dynamic condition handling
- **Events** — cart lifecycle and merge events documented in the deeper package pages
- **Persistence** — owner-aware cart records with optional Signals enrichment

## Owner scoping and security notes

- Cart owner scoping uses `owner_type` and `owner_id`; derived keys such as `owner_scope` are internal implementation details, not authorization boundaries
- Consuming packages should still validate ownership server-side when mutating carts or converting them into downstream checkout or order records

`aiarmada/cart` is the lightweight cart foundation for the Commerce monorepo. It owns cart storage, cart items, conditions, metadata, login migration, events, and owner-aware persistence.

## What stays in cart

- Database-backed cart storage with optimistic locking.
- Cart item, condition, and metadata APIs.
- Dynamic condition support.
- Cart lifecycle events such as item added, item removed, cart cleared, and cart merged.
- Optional owner scoping through `commerce-support` using `owner_type` and `owner_id`.

## Optional intelligence

Cart does not store recovery campaigns, daily metrics, popup interventions, or alert rules. Those concerns belong to `aiarmada/signals` when analytics and alerting are needed.

Install and enable Signals to capture cart events as analytics/alert input:

```bash
composer require aiarmada/signals
```

Then enable the relevant Signals integration in `config/signals.php`.

## Multitenancy

Cart owner scoping uses `owner_type` and `owner_id`. Internal uniqueness keys such as `owner_scope` are generated by `commerce-support` helpers and are not public API, export data, or authorization boundaries.
